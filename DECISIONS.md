## ADR-001. ORM — TypeORM (а не Prisma)
- **Дата:** 2026-07-04
- **Контекст:** NestJS + Postgres, нужны миграции и естественная интеграция с DI.
- **Решение:** TypeORM (decorator-based entities, встроенные миграции).
- **Альтернативы:** Prisma — сильная типизация, но отдельная schema-DSL вне NestJS-стиля и иная модель интеграции.
- **Последствия:** Entities и репозитории в идиоматике NestJS; миграции как аналог Alembic из Mini-SRM. Ближе к уже показанному опыту.

## ADR-002. Форматы прайсов v1 — CSV + XLSX (PDF вне границ)
- **Дата:** 2026-07-04
- **Контекст:** Поставщики шлют прайсы в разных форматах; PDF-таблицы без границ — самая тяжёлая часть.
- **Решение:** В v1 — CSV и XLSX. PDF — отдельный stretch.
- **Альтернативы:** сразу +PDF (раздувает объём и тесты); только CSV (слабее демонстрирует нормализацию).
- **Последствия:** Достаточно «грязи» для демонстрации нормализации (кодировки, плавающие шапки), без ада PDF-парсинга. Границы явные.

## ADR-003. Матчинг — артикул (точный) → fuzzy по названию
- **Дата:** 2026-07-04
- **Контекст:** Позицию нужно сопоставить с эталонной номенклатурой.
- **Решение:** Сначала точный матч по нормализованному артикулу; при отсутствии — fuzzy по нормализованному названию (коэффициент Дайса) с порогами HIGH/LOW; спорное — на ручной разбор.
- **Альтернативы:** только fuzzy по названию (проще, но менее точно); ML-классификатор (нужен датасет).
- **Последствия:** Детерминированно и объяснимо (легко тестировать и обосновать confidence). Пороги — через env.

## ADR-004. Интеграция с Mini-SRM — экспорт-эндпоинт (а не webhook)
- **Дата:** 2026-07-04
- **Контекст:** Price-Normalizer должен отдавать чистый каталог системе закупок.
- **Решение:** `GET /export/catalog?format=json|csv` — pull-модель.
- **Альтернативы:** push/webhook в Mini-SRM — реалистичнее, но требует согласования двух сервисов и усложняет тесты/демо.
- **Последствия:** Просто, декларативно, легко проверяется в тестах и описывается в README. При необходимости push добавляется поверх.

## ADR-005. Очередь фоновой обработки — BullMQ
- **Дата:** 2026-07-04
- **Контекст:** Парсинг+нормализация+матчинг тяжёлые, нужны фоново; Redis уже в стеке.
- **Решение:** BullMQ (стандарт для Node поверх Redis) + отдельный worker-процесс/контейнер.
- **Альтернативы:** обработка в HTTP-запросе (блокирует, таймауты); самописная очередь (лишний код).
- **Последствия:** Аналог RQ из Mini-SRM; батч двигается по стадиям через ту же таблицу переходов.

## ADR-006. Native-free зависимости: bcryptjs + sql.js (вместо bcrypt + better-sqlite3)
- **Дата:** 2026-07-04
- **Контекст:** На машине разработки нет toolchain для нативной сборки (node-gyp/MSVC), а Node 24 очень свежий — у нативных пакетов часто нет prebuild'ов. `better-sqlite3` упал на `node-gyp rebuild`; `bcrypt` — тоже нативный и упал бы так же на шаге 3.
- **Решение:** тесты на **sql.js** (WASM-sqlite, TypeORM `type: 'sqljs'`), хеширование на **bcryptjs** (чистый JS). Весь стек стал собираться без компиляторов.
- **Альтернативы:** ставить build tools / Postgres для тестов — тяжелее и менее переносимо (CI, чужие машины).
- **Последствия:** Установка проходит везде без нативной сборки. bcryptjs медленнее нативного bcrypt, но для pet-проекта несущественно; API совместим.

## ADR-007. PK — integer (отклонение от SPEC «bigint»)
- **Дата:** 2026-07-04
- **Контекст:** SPEC §3 указывал bigint. Но в TypeORM `bigint` возвращается в JS как **строка** (усложняет DTO/фронт/матчинг), а на sqlite (тесты) авто-инкремент нормально работает только с `integer`.
- **Решение:** `@PrimaryGeneratedColumn()` (integer) для всех PK. Отклонение от SPEC осознанное и зафиксировано здесь.
- **Последствия:** id — обычные JS-числа, чище DTO и фронт. Диапазона integer с запасом хватает для pet-проекта; при реальном масштабе — вернуть bigint с строковым id.

## ADR-008. Слои API: Nest-исключения, единый transition-эндпоинт, синхронная обработка (пока)
- **Дата:** 2026-07-04
- **Контекст:** Нужны осмысленные HTTP-коды, строгая валидация переходов стадий и рабочий пайплайн до введения очереди (шаг 5).
- **Решения:**
  - Доменные ошибки — через встроенные Nest-исключения (`ConflictException`→409, `ForbiddenException`→403, `UnprocessableEntityException`→422, `NotFoundException`→404). Nest сам маппит их в JSON `{message, statusCode}` — не изобретаем свой слой ошибок.
  - Переходы стадий — декларативная таблица (`common/pipeline/batch-transitions.ts`); `BatchesService.applyTransition` делает lookup, недопустимая пара → 409. HTTP-эндпоинт `/batches/:id/transition` разрешает только `actor:'operator'`-переходы (`finish_review`,`retry`); worker-переходы через HTTP → 403.
  - Глобальные guard'ы (`APP_GUARD`): JWT везде, кроме `@Public()` (`/auth/login`, `/health`); роли через `@Roles()`.
  - **Обработка батча выполняется синхронно** в `BatchProcessorService.process()`, вызывается из upload-контроллера. На шаге 5 триггер переносится в BullMQ-воркер — сам `process()` не меняется (файл кладётся в стор, воркер зовёт тот же метод).
- **Проверено:** typecheck + eslint чисто; runtime-smoke на sql.js прошёл весь цикл: login → upload CSV (2 строки, ; + русские заголовки) → 1 авто-матч по артикулу + 1 needs_review → недопустимый `finish_review` = 409 → resolve → авто-completed → export (1 позиция).

## ADR-009. Выравнивание со SPEC по итогам тестов: 422 на валидацию, failed восстановим
- **Дата:** 2026-07-04
- **Контекст:** Тесты выявили два расхождения кода и SPEC.
- **Решения:**
  - **Валидация DTO → 422.** NestJS `ValidationPipe` по умолчанию отдаёт **400**, а SPEC §9 требует 422 для ошибок валидации. Задал `errorHttpStatusCode: 422` (и в `main.ts`, и в тестовой фабрике) — теперь ошибки валидации и бизнес-правил единообразно 422.
  - **`failed` — не терминальный.** SPEC §5.1 в тексте называл `failed` терминальным, но §5.2 определяет переход `retry: failed → uploaded`. Значит `failed` восстановим. `TERMINAL_STATES` выводится из таблицы и содержит только `completed`. Тест приведён к реальной (и более полезной) семантике: из `failed` доступен `retry`.
- **Последствия:** HTTP-коды соответствуют SPEC §9; батч после сбоя можно перезапустить. Мелкая текстовая неточность SPEC §5.1 (про терминальность failed) зафиксирована здесь.

## ADR-010. Redis: graceful degradation, опциональная очередь BullMQ, guard rate-limit
- **Дата:** 2026-07-04
- **Контекст:** Redis должен нести кэш каталога, фоновую обработку батча и rate limiting (SPEC §7), но тесты обязаны оставаться герметичными (без внешнего Redis).
- **Решения:**
  - **`RedisService`** (тонкая обёртка ioredis, `@Global`): при `DISABLE_REDIS=true` (тесты) или ошибке соединения — деградирует (get→null, rate-limit→allow). Каталог не падает без Redis.
  - **Кэш каталога** в `ProductsService`: ключи `catalog:list:*`/`catalog:item:*`/`catalog:active`, TTL 300; любая мутация → `delByPattern('catalog:*')`. Матчер читает `catalog:active` из кэша.
  - **Очередь BullMQ регистрируется условно** (`REDIS_ENABLED`): в тестах модуль очереди не импортируется, `BatchDispatcher` через `@Optional() @InjectQueue` получает `undefined` и обрабатывает **инлайн**; в проде — кладёт джобу, воркер (`ImportsProcessor`) её берёт. Файл передаётся в payload джобы как **base64** (без общего тома между контейнерами).
  - **Rate limiting** — собственный `RateLimitGuard` (APP_GUARD) + декоратор `@RateLimit(limit, window)` поверх `INCR`+`EXPIRE`. Fail-open при недоступности Redis. Применён к `login` (10/60) и загрузке батча (5/60).
  - Upload теперь возвращает **202** (SPEC §8): создаёт батч, диспатчит обработку, отдаёт текущее состояние.
- **Последствия:** Redis задействован по всем трём пунктам; отказ Redis не роняет API; тесты полностью герметичны (71 тест на sqljs без Redis). E2E round-trip реальной очереди проверяется на шаге 7 (docker).

## ADR-011. Фронтенд: React + TS + TanStack Query, экран разбора матчей
- **Дата:** 2026-07-04
- **Контекст:** Нужен минималистичный UI под роли operator/viewer: загрузка прайсов, разбор спорных матчей, каталог, сравнение цен (SPEC §6).
- **Решения:**
  - Стек: React 18 + TS strict + Vite + TanStack Query + react-router. `fetch`-обёртка с типизированным `ApiError` (без axios).
  - Страницы: `BatchesPage` (загрузка + список со `refetchInterval`, чтобы отражать прогресс фонового воркера), `BatchReviewPage` (разбор `needs_review`-позиций: кандидаты с confidence → «Выбрать»/«Новый товар»/«Отклонить»), `CatalogPage` (номенклатура + поставщики + сравнение цен по товару).
  - Auth: JWT в localStorage, `AuthContext`, защищённые маршруты; действия operator скрыты для viewer.
- **Проверено:** `npm run build` (tsc strict + vite) и `npm run lint` (eslint 9) — чисто; dev-сервер поднимается, страница входа рендерится (скрин в `docs/screenshots/login.png`). Полные CRUD/разбор-флоу — на шаге 7 с живым бэкендом.

## ADR-012. Docker Compose: 5 сервисов, миграции на старте, выделенный worker
- **Дата:** 2026-07-04
- **Контекст:** `docker compose up` должен поднимать всю систему одной командой (SPEC шаг 7).
- **Решения:**
  - Сервисы: `db` (postgres:16), `redis` (redis:7, healthcheck), `api`, `worker`, `web`.
  - **api** (multi-stage `node build → node прод-образ`): на старте `node dist/seed.js && node dist/main.js`. Миграции применяются автоматически (`migrationsRun: true` + `migrations: dist/database/migrations/*.js`), сид идемпотентный.
  - **worker** — тот же образ, `WORKER=true`, команда `node dist/worker.js` (context без HTTP). `ImportsProcessor` регистрируется только при `WORKER=true`, поэтому api кладёт джобы, а потребляет их отдельный воркер.
  - **web** — multi-stage `node build → nginx` c SPA-fallback; `VITE_API_BASE_URL` вшивается build-arg'ом.
  - Секреты — из `.env` (`${JWT_SECRET:?}`), `.env` в `.gitignore`, есть `.env.example`.
- **Проверка:** `docker compose config` валиден (5 сервисов, интерполяция env, healthcheck'и, контексты сборки). Реальный `docker compose build/up` на момент шага не выполнялся — демон Docker Desktop не был запущен; запуск стека одной командой — за пользователем (инструкция в README). Функционально всё проверено без Docker: 71 тест на sqljs + сборка фронта.

## Инциденты (переписанный код / баги / уязвимости)

### INC-001. Nullable-колонки с union-типом → DataTypeNotSupportedError
- **Симптом:** при построении схемы на sql.js — `Data type "Object" in "CanonicalProduct.article" is not supported`.
- **Причина:** для поля `article: string | null` reflect-metadata отдаёт design-тип `Object` (union не резолвится), а `@Column` без явного `type` пытается вывести тип из рефлексии → «Object».
- **Исправление:** добавил явный `type` всем nullable-колонкам с union-типом (`type: 'varchar'` для строк, `type: 'int'` для чисел): `Supplier.contactPerson/email`, `CanonicalProduct.article`, `PriceOffer.rawArticle/canonicalProductId/reviewedById`.
- **Урок:** в TypeORM у любой `T | null`-колонки тип нужно задавать явно — на не-null полях вывод из рефлексии работает, на union'ах молча ломается. Ловится только реальной инициализацией DataSource, не компиляцией TS.
